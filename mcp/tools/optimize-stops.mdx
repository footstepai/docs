---
title: 'optimize_stops'
description: 'Find the fastest order to visit multiple stops'
---

Given a list of stops, find the most efficient order to visit them all. Solves the travelling salesman problem and returns the optimised route with full terrain analytics, plus a breakdown of how much distance and time you saved compared to the original order.

## Example prompts

- "Find the best order to visit these 8 delivery addresses"
- "I need to visit 5 clients today, what's the most efficient route?"
- "Optimise this list of warehouse stops for a bike courier"
- "Reorder these 12 locations to minimise driving time"

## What you get back

The optimised route (same detail as `get_directions`: distance, duration, steps, terrain) plus the reordered stop sequence and savings. Savings show distance saved in meters and as a percentage, and time saved in seconds and as a percentage.

## Rendering on a map: `include_geometry`

By default this tool returns a lean response — the encoded polyline of the optimised route and the sample-by-sample elevation profile are stripped out so standalone LLM agents (which can't render maps) don't pay for them in tokens.

If your client renders results on a map (Mapbox, MapLibre, Leaflet, etc.), pass `include_geometry: true`. The response then carries the route polyline and elevation profile per leg alongside the same summary fields. The map renders the geometry; the LLM still reasons over the summary.

Setting `include_geometry: true` on every call from a map-rendering client is tedious. See the [getting-started guide](/mcp/getting-started#map-rendering-clients) for a one-line Vercel AI SDK middleware that injects the flag automatically.

Run the same route with different travel modes and get a side-by-side comparison of distance, duration, and terrain difficulty. Useful for deciding how to travel.

Supports comparing 2-4 modes in one call: walking, cycling, driving, bus, and truck.

## Example prompts

- "Should I walk or cycle from Kings Cross to Tower Bridge?"
- "Compare driving vs cycling from my hotel to the conference centre"
- "What's the easiest way to get from A to B - walking, cycling, or driving?"
- "How much hillier is cycling vs walking between these two points?"

## What you get back

A route for each travel mode (with full distance, duration, terrain analytics, and turn-by-turn steps) plus a comparison showing which mode is fastest, shortest, and easiest by terrain. Includes a natural language summary like "Cycling is 3 min faster than walking but has 40m more climbing."

## Rendering on a map: `include_geometry`

By default this tool returns a lean response — each per-mode polyline and the sample-by-sample elevation profile are stripped out so standalone LLM agents (which can't render maps) don't pay for them in tokens.

If your client renders results on a map (Mapbox, MapLibre, Leaflet, etc.), pass `include_geometry: true`. The response then carries the route polyline and elevation profile per mode alongside the same summary fields. The map renders the geometry; the LLM still reasons over the summary.

Setting `include_geometry: true` on every call from a map-rendering client is tedious. See the [getting-started guide](/mcp/getting-started#map-rendering-clients) for a one-line Vercel AI SDK middleware that injects the flag automatically.
